# 第3章 需求分析

## 3.1 引言

剧本杀预约与门店管理系统是一个面向多类用户的综合性信息平台，旨在为用户提供便捷的剧本杀游戏预约体验，同时为门店管理者提供高效的经营管理工具。系统的设计需要充分满足不同用户角色的需求，包括普通用户、系统管理员、门店管理员和门店员工。为了确保系统的实用性和可行性，需要对系统的功能需求和性能需求进行深入分析。本章将从功能需求和非功能需求两个方面，对剧本杀预约与门店管理系统的各项需求进行系统阐述，为系统设计和开发提供清晰的指导方向。

## 3.2 系统功能需求

### 3.2.1 登录注册

用户登录注册是系统的基础功能，直接关系到用户的身份识别和数据安全。系统应提供基于手机号和密码的注册方式，允许用户通过输入有效的手机号、设置安全密码来创建新账户。为了防止恶意注册和自动化攻击，系统在注册过程中集成了滑块验证码机制，确保注册操作由真实用户完成。登录功能应支持手机号与密码的组合验证，并通过滑块验证码提升安全级别。系统需要记录用户的登录状态，为后续的权限控制和个性化服务提供基础。此外，系统应提供密码重置功能，允许用户通过手机号验证来恢复忘记的密码，确保用户的账户安全性。

### 3.2.2 个人信息管理

个人信息管理模块允许用户维护和更新自身的账户信息。用户可以修改的信息包括昵称、头像、手机号、邮箱、生日和实名认证等字段。这些信息的完整性和准确性对于系统的后续功能（如会员等级判定、生日礼券发放、订单管理等）具有重要意义。系统应提供直观的信息编辑界面，允许用户随时更新个人信息。对于敏感信息如手机号和邮箱，系统应实现验证机制，确保修改操作的合法性。实名认证功能是系统的重要安全保障，用户在进行大额消费或获取特定权限时可能需要提供实名信息。系统应安全存储这些个人信息，并遵守数据保护相关法律法规。

### 3.2.3 剧本管理

剧本管理功能是系统的核心，为用户提供丰富的剧本浏览和查询体验。系统应支持多维度的剧本展示，包括列表视图和详细信息视图。用户可以按照分类、标签等多种方式对剧本进行筛选，快速定位感兴趣的剧本。系统需要为每个剧本提供详细的信息展示，包括剧本名称、风格、难度、时长、人数要求、价格等关键信息。剧本应支持多层级的分类体系，如按游戏风格（推理、悬疑、喜剧等）分类。系统应实现强大的搜索功能，允许用户通过关键词快速定位所需剧本。为了提升用户体验，系统应支持剧本收藏功能，用户可以将感兴趣的剧本加入收藏夹以便后续查看。系统还应记录用户的浏览历史，为个性化推荐提供数据支持。

### 3.2.4 门店管理

门店管理功能为用户提供门店的查询和定位服务，为门店管理者提供门店信息维护的工具。用户可以浏览门店列表，查看各门店的详细信息，包括门店名称、地址、联系方式、营业时间、设施信息等。系统应集成地图定位功能，允许用户查看门店的地理位置，并基于地理信息进行门店筛选和排序。系统应支持用户对门店的收藏功能，便于用户快速访问常用门店。对于门店管理者，系统应提供门店信息维护工具，允许其更新门店的基本信息、营业时间、联系方式等内容。门店管理者可以通过系统查看本门店的经营看板，包括日度、周度、月度的营销数据和关键指标，为经营决策提供数据支持。

### 3.2.5 在线预约

在线预约是系统的关键业务功能，涉及订单创建、支付和核销等多个环节。用户在预约时需要依次选择剧本、门店、游戏场次、参加人数，然后选择可用的优惠券以获得折扣。系统应实时展示每个场次的座位可用情况，确保用户不会预约已满的场次。用户可以通过支付宝支付功能完成订单支付，系统应与支付宝进行完整的支付流程集成，包括订单创建、支付确认和支付结果回调。预约成功后，系统应生成包含核销码的订单凭证，用户可以通过核销码在门店完成核销。系统应支持订单管理功能，用户可以查看所有订单状态、取消订单、申请改期或退款。改期功能应允许用户将已预约的场次更改为其他时间段的场次，需要确保新选择的场次有足够的座位。

### 3.2.6 VIP会员

VIP会员功能为系统引入了差异化的用户权益体系，旨在提升用户粘性和消费金额。系统应设计四个等级的VIP会员体系，不同等级对应不同的会员权益。用户可以通过购买会员卡的方式升级会员等级，系统应支持会员续费功能，允许用户在会员过期前进行续费以保持会员身份。不同等级的会员应获得相应的会员权益，包括月度体验券、生日礼券和积分倍率等。月度体验券是专属于VIP会员的优惠券，用户每个月可以获得特定金额的体验券，用于抵扣预约费用。生日礼券是系统在用户生日月份发放的特殊权益，旨在增强用户的归属感。积分倍率权益使VIP会员在消费时能够获得更多的积分，提高积分的积累速度。

### 3.2.7 优惠券与积分

优惠券和积分构成了系统的营销和激励体系。优惠券功能允许用户通过多种方式获取优惠券，包括直接领取系统发放的优惠券、通过积分兑换优惠券等方式。系统应为每个优惠券设置使用条件，如最低消费金额、适用的剧本或门店范围等，确保优惠券的有效使用。用户可以在预约时选择可用的优惠券抵扣部分费用。系统应清晰展示每个优惠券的使用状态，如未使用、已使用、已过期等。积分系统为用户提供了多种获得积分的途径，包括签到获积分、完成订单获积分、参与游戏活动获取积分等。系统应记录用户的积分余额，允许用户使用积分兑换优惠券或参加特定的权益活动。积分的获取比例和兑换比例应由系统管理员统一配置，以支持灵活的营销策略。

### 3.2.8 拼团功能

拼团功能是系统的社交互动特性，通过降低用户的消费成本来促进用户参与。用户可以发起拼团，选择指定的剧本、门店和场次，设定拼团所需的人数。发起拼团后，系统应为拼团生成唯一的拼团码，用户可以通过分享拼团码邀请其他用户加入拼团。其他用户可以通过拼团码或扫描拼团二维码来查看拼团详情，并决定是否加入。系统应实时展示拼团的参加人数和拼团进度。当拼团人数达到目标人数后，拼团自动成团，所有参加拼团的用户都能够获得相应的拼团折扣。拼团成团前，用户可以选择退出拼团，系统应及时更新拼团状态。系统应设置拼团的有效期，超时未成团的拼团应自动关闭，参加拼团的用户应获得退款。

### 3.2.9 AI客服

AI客服功能通过集成大语言模型为用户提供24小时的智能客服服务。系统采用DeepSeek大模型作为核心的语义理解和回复生成引擎，结合本地知识库实现高质量的客服回复。用户在使用系统过程中遇到问题时，可以通过AI客服进行提问，系统应快速处理用户的自然语言输入，理解用户的真实需求。AI客服应能够回答关于系统功能、预约流程、会员权益等常见问题，并提供相应的解决方案。系统应维护一个持续更新的本地知识库，包含常见问题解答、使用指南、政策说明等内容，当DeepSeek模型的回复不够准确或信息过时时，知识库可以作为兜底方案，确保用户获得正确的信息。对于AI客服无法解决的复杂问题，系统应支持平滑的人工客服转接，确保用户的问题最终能够得到妥善处理。

### 3.2.10 评价与反馈

评价与反馈功能为系统提供了用户意见的收集机制，有助于平台和门店持续改进。用户在完成一次预约后，系统应邀请用户提交评价。用户可以对本次游戏体验进行评分，评分通常采用五星制或其他量化评分体系。除了评分外，用户还可以提交文字评价，详细描述游戏体验、服务质量、场景环境等方面的意见。系统应为不同的评价因素提供独立的评分维度，如游戏设计、主持人表现、场景环境、服务态度等，以收集更详细的反馈信息。系统应展示评价的统计汇总，包括平均评分、评分分布等，为其他用户的选择提供参考。评价内容应经过审核，确保评价内容的合理性和真实性。此外，用户可以向平台提交其他类型的反馈和建议，系统应提供反馈收集和处理的完整流程，允许管理员对反馈进行分类、处理和回复。

### 3.2.11 通知管理

通知管理功能为系统的各个业务场景提供消息推送支持。系统应支持多种类型的通知，包括订单相关通知（预约确认、支付成功、改期提醒、取消确认等）、会员相关通知（等级升级、权益过期、礼券发放等）、系统公告和活动推送等。用户应在系统设置中能够配置通知偏好，选择接收哪些类型的通知，以及通知的接收方式（如应用内通知、短信、邮件等）。系统应在适当的时机自动发送相关通知，确保用户及时了解订单状态和系统动态。对于门店管理者和员工，系统应推送门店相关的通知，如新订单提醒、预约变更通知等。系统应提供通知历史查看功能，用户可以查看过去接收的所有通知，便于查询历史信息。

### 3.2.12 系统管理员功能

系统管理员拥有最高权限，负责整个系统的平台级管理工作。系统应为管理员提供用户管理功能，包括用户信息查看、用户账户禁用、用户数据统计等操作。门店管理功能允许管理员查看和维护所有门店的信息，审核新门店的加入申请，设置门店的上线和下线状态。剧本管理功能允许管理员创建和维护系统中的剧本信息，包括剧本的基本信息、价格、可用门店等内容，管理员可以对剧本进行上下架操作，控制剧本的可见性。员工管理功能使管理员能够查看和管理系统中的所有员工账户，包括员工的角色分配、权限设置等。通知公告功能允许管理员发布系统级的通知和公告，向所有或指定的用户推送消息。反馈处理功能为管理员提供了反馈的集中管理工具，管理员可以查看用户提交的反馈，进行分类、分配、处理和回复。优惠券管理功能允许管理员创建和维护优惠券，包括优惠券的折扣金额、使用条件、有效期等。VIP套餐管理功能允许管理员配置VIP会员的等级和权益，设置各等级的价格和权益内容。此外，系统应为管理员提供数据统计功能，展示系统的关键指标，如日活跃用户数、订单总数、营收数据等，为经营决策提供数据支持。

### 3.2.13 门店管理员功能

门店管理员负责本门店的日常经营管理工作，拥有本门店的最高权限。门店信息维护功能允许门店管理员更新本门店的基本信息，包括门店名称、地址、联系方式、营业时间、设施介绍等内容。经营看板功能为门店管理员提供了本门店的经营数据展示，包括日度、周度、月度的订单数量、营收数据、预约率等关键指标，帮助管理员实时掌握门店的经营状况。DM管理功能允许门店管理员维护本门店的主持人（DM）信息，包括DM的基本信息、工作状态、档期等。员工管理功能允许门店管理员添加和维护本门店的员工账户，配置员工的角色和权限，实现对员工的有效管理。预约管理功能为门店管理员提供了对预约订单的集中管理，管理员可以查看所有预约，处理退款申请，管理预约的核销状态。排期管理功能允许门店管理员制定本门店的游戏场次安排，包括设置每个剧本的可用场次、时间、可容纳人数等信息，为用户的在线预约提供基础。每日报表功能为门店管理员生成日度的经营报表，包括当日的订单数、营收、核销率等数据，便于管理员进行日常经营分析。

### 3.2.14 门店员工功能

门店员工根据角色不同拥有不同的权限。系统定义了三种员工角色：MANAGER（店长或副店长）、DM（主持人）和CLERK（服务员）。MANAGER角色拥有最高的员工权限，可以查看所有预约订单、进行预约的核销操作、管理DM的分配、处理退款申请、管理其他员工账户、查看和生成员工报表。DM角色是主持人，可以查看分配给自己的预约，标记场次为已完成，查看与预约相关的通知。CLERK角色是服务员，可以查看预约订单、进行预约的核销操作，以及查看相关的系统通知。这种分层的权限设计确保了不同角色的员工只能访问和操作其工作职责范围内的信息和功能，提高了系统的安全性和操作的便利性。

## 3.3 系统性能需求

### 3.3.1 安全性

安全性是系统的关键非功能性需求。系统应采用JWT（JSON Web Token）作为主要的身份验证和授权方案，用户登录成功后获得JWT令牌，后续的请求需要在HTTP头中携带有效的令牌才能被系统接受。系统应实现细粒度的接口权限控制，确保每个API接口都被正确的权限保护，不同角色的用户只能访问其权限范围内的接口。为了防止非法用户的自动化攻击，系统在用户注册阶段集成了滑块验证码机制，要求用户完成人类验证才能成功注册，有效防止机器人的恶意注册。系统采用角色和权限码的双层控制方案，首先通过用户的角色确定其基本权限范围，然后通过更细粒度的权限码进行二次权限验证，确保权限控制的准确性和安全性。敏感信息如用户密码应在数据库中进行加密存储，不能明文保存。用户的个人信息应按照数据保护法规要求进行安全存储和传输，确保用户隐私不被泄露。

### 3.3.2 高效性

高效性需求涉及系统的性能、响应速度和用户体验。系统应采用Redis缓存来提升数据访问速度，对于频繁访问的数据如剧本列表、门店信息、用户会员信息等，应存储在Redis缓存中，减少数据库的查询压力，加速系统的响应速度。系统采用前后端分离的架构设计，前端和后端分别进行独立开发和部署，前端通过REST API与后端进行通信。这种架构设计使得前端应用可以独立优化，采用现代化的前端框架进行性能优化，同时后端可以专注于业务逻辑和数据处理的优化，提高系统的整体性能。系统应支持WebSocket实时推送功能，当出现需要实时通知用户的事件时（如新订单、消息提醒、人工客服消息等），系统可以通过WebSocket主动向客户端推送消息，避免客户端频繁的轮询请求，减少服务器的负担，提升系统的实时性和高效性。数据库查询应进行优化，使用适当的索引和查询优化技术，确保常见查询的高效执行。系统应进行容量规划，确保服务器有足够的计算资源和存储空间满足预期的并发用户数和数据增长。
