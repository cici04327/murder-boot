## 2. 项目整体实现结构

### 2.1 总体架构

该项目采用的是 **Spring Boot 单体后端 + Vue3 双前端** 的实现方式。

- 后端：`src/main/java/com/murder`
- 用户端前端：`frontend/user`
- 管理端前端：`frontend/admin`
- 数据库脚本：`sql/`
- Spring Boot 静态资源托管目录：`src/main/resources/static`、`src/main/resources/static/admin`

它的特点不是“只做信息展示”，而是做了一个完整的剧本杀门店经营闭环系统，主要包括：

- 剧本展示
- 门店展示
- 在线预约
- 在线支付
- 退款处理
- 门店员工协同
- 会员与优惠券
- AI 客服
- 人工客服
- AI 排班
- 通知与消息推送

### 2.2 后端核心技术

后端在 `pom.xml` 中引入并组合了这些主要技术：

- Spring Boot
- MyBatis-Plus
- MySQL
- Redis
- JWT
- WebSocket
- Knife4j / OpenAPI
- 支付宝 Java SDK
- Jackson
- Druid 数据源

启动类：

- `src/main/java/com/murder/MurderBootApplication.java`

从启动注解可以看出，这个项目除了提供接口以外，还启用了：

- Mapper 扫描
- 缓存
- 定时任务
- 事务管理

所以它不是简单 CRUD，而是有状态流转、消息、支付与定时处理逻辑。

### 2.3 前端核心技术

两个前端都采用：

- Vue 3
- Vite
- Element Plus
- Axios
- Vue Router

用户端位于：

- `frontend/user`

管理端位于：

- `frontend/admin`

这两套前端分别服务于不同角色：

- 用户端：普通用户下单、查看剧本、支付、评价、咨询
- 管理端：总部、门店管理员、员工/DM 进行运营与履约管理

## 3. 后端代码分层是怎么实现的

项目后端整体采用比较典型的分层结构：

### 3.1 controller 层

负责接收 HTTP 请求、参数校验、角色范围控制、返回统一结果。

典型控制器包括：

- `AIController`
- `AiScheduleController`
- `ReservationController`
- `PaymentController`
- `VipController`
- `StoreController`
- `StoreEmployeeController`
- `ServiceSessionController`

### 3.2 service 层

负责实现具体业务逻辑，是系统最核心的部分。

典型实现类包括：

- `AIServiceImpl`
- `AiScheduleServiceImpl`
- `ReservationServiceImpl`
- `PaymentServiceImpl`
- `VipServiceImpl`
- `RecommendationServiceImpl`
- `ServiceSessionServiceImpl`

### 3.3 mapper 层

负责数据库访问，使用 MyBatis-Plus。

例如：

- `ReservationMapper`
- `ScriptScheduleMapper`
- `DmMapper`
- `AiScheduleTaskMapper`

### 3.4 entity / dto / vo

- `entity`：数据库实体
- `dto`：前端传入的请求对象
- `vo`：返回给前端的视图对象

这是比较标准的企业项目写法，便于前后端解耦。

## 4. 前端是怎么与后端对接的

### 4.1 统一请求封装

用户端与管理端都对 Axios 做了统一封装：

- 用户端：`frontend/user/src/utils/request.js`
- 管理端：`frontend/admin/src/utils/request.js`

统一封装主要做了几件事：

1. 统一 `baseURL: /api`
2. 自动携带 token
3. 自动处理 401/403
4. 管理端自动打 `X-Client-Type: admin`
5. 门店管理员/员工请求会补充 `X-Store-Id`
6. 某些 GET/POST 请求会自动附带 storeId

### 4.2 API 文件与页面目录一一对应

例如用户端：

- `frontend/user/src/api/ai.js` 对应 AI 客服
- `frontend/user/src/api/reservation.js` 对应预约/支付/退款
- `frontend/user/src/api/store.js` 对应门店

管理端：

- `frontend/admin/src/api/ai-schedule.js` 对应 AI 排班
- `frontend/admin/src/api/store-employee.js` 对应员工管理
- `frontend/admin/src/api/notification.js` 对应通知中心

这种写法的优点是前端结构很清楚，查问题时可以从页面直接找到 API 文件。

## 5. 鉴权与权限是怎么实现的

### 5.1 JWT 认证机制

项目的认证核心在：

- `src/main/java/com/murder/interceptor/JwtTokenInterceptor.java`
- `src/main/java/com/murder/common/properties/JwtProperties.java`
- `src/main/java/com/murder/common/context/BaseContext.java`

请求进入后端时，JWT 拦截器会：

1. 从请求头取 token
2. 根据请求类型判断使用用户端密钥还是管理端密钥
3. 解析 claims
4. 将用户信息写入 `BaseContext`

写入的上下文包括：

- 当前用户 ID
- 角色
- storeId
- employeeId
- staffRole
- permissionCodes
- dmId

后续 Controller 和 Service 就可以通过 `BaseContext` 获取当前登录人身份。

### 5.2 管理端与用户端的区别

管理端请求会自动带：

- `token`
- `X-Client-Type: admin`

用户端默认按普通用户处理。

### 5.3 权限模型

角色并不是简单的“管理员/用户”二分，而是至少包含：

- `USER`
- `SUPER_ADMIN`
- `STORE_ADMIN`
- `STORE_STAFF`

员工端还支持更细的权限码，例如：

- `employee:manage`
- `report:view`
- `notification:view`
- `reservation:checkin`
- `reservation:complete`
- `reservation:assign_dm`
- `refund:process`

这意味着系统在实现上不是只靠前端隐藏菜单，而是后端也会做严格权限校验。

## 6. 支付模块是怎么实现的

支付相关实现集中在：

- 控制器：`src/main/java/com/murder/controller/PaymentController.java`
- 服务：`src/main/java/com/murder/service/impl/PaymentServiceImpl.java`
- 配置：`src/main/java/com/murder/common/config/AlipayConfig.java`
- 开发环境配置：`src/main/resources/application-dev.yml`
- 用户端支付页：`frontend/user/src/views/payment/index.vue`
- 用户端支付结果页：`frontend/user/src/views/payment/result.vue`
- 用户端 API：`frontend/user/src/api/reservation.js`

### 6.1 支付接口入口

支付控制器 `PaymentController` 暴露了这些主要接口：

- `POST /api/reservation/payment/create`：创建支付订单
- `POST /api/reservation/payment/notify`：支付宝异步通知
- `GET /api/reservation/payment/return`：支付宝同步回跳
- `GET /api/reservation/payment/status/{reservationId}`：查询支付状态
- `POST /api/reservation/payment/refund/apply`：用户申请退款
- `POST /api/reservation/payment/refund/process`：管理端处理退款

### 6.2 创建支付的实现过程

用户在支付页点击“确认支付”后：

1. 前端页面 `frontend/user/src/views/payment/index.vue` 调用
   - `createPayment(reservationId, 'alipay')`
2. 用户端 API `frontend/user/src/api/reservation.js` 向后端发送：
   - `POST /reservation/payment/create`
3. `PaymentController.createPayment(...)` 先校验预约是否属于当前用户
4. 校验通过后调用 `paymentService.createPayment(...)`
5. 服务层构造支付宝支付请求并返回支付表单 HTML
6. 前端将后端返回的 HTML 直接提交到支付宝页面完成付款

这里的关键点是：

- 支付不是前端直接拼支付宝地址
- 而是由后端生成符合支付宝要求的支付表单
- 用户支付完成后再由支付宝回调后端

### 6.3 为什么要有同步回跳和异步通知

这是支付宝接入的标准方式：

#### 同步回跳 `return`
用户支付后，浏览器跳回系统后端，再由后端重定向到用户端的结果页。

用途：

- 给用户一个“支付成功/失败”的页面反馈
- 页面地址一般会带上 `success`、`reservationId`、`message` 等参数

#### 异步通知 `notify`
这是支付宝服务器直接通知本系统后端。

用途：

- 以支付宝服务器通知结果作为最终支付确认依据
- 更新订单支付状态
- 避免用户关闭浏览器导致系统状态不同步

所以从系统可靠性角度看，**异步通知才是最终入账依据，同步回跳只是前端体验层结果展示**。

### 6.4 支付宝沙箱是怎么接入的

支付宝接入配置类在：

- `src/main/java/com/murder/common/config/AlipayConfig.java`

它通过 `@ConfigurationProperties(prefix = "alipay")` 读取配置项，并创建：

- `AlipayClient`

典型配置包括：

- `appId`
- `privateKey`
- `publicKey`
- `notifyUrl`
- `returnUrl`
- `resultUrl`
- `gatewayUrl`
- `charset`
- `format`
- `signType`
- `mockPayment`

开发环境里：

- `application-dev.yml` 中配置了沙箱环境参数
- `gateway-url` 指向支付宝沙箱网关
- `notify-url` 与 `return-url` 配置为后端地址
- `result-url` 配置为用户端支付结果页地址

这套设计说明项目在开发时采用了：

- 后端接沙箱网关
- 前端只负责发起支付和展示支付结果
- 回调都由后端负责处理

### 6.5 退款是怎么实现的

退款逻辑主要在 `PaymentServiceImpl` 中。

实现特点：

1. 用户先提交退款申请
2. 后台审核后调用处理退款逻辑
3. 如果不是 mock 支付模式，就调用支付宝退款 API
4. 成功后更新预约退款状态、支付状态，并推送通知

从实现细节上看，服务层已经调用了：

- `AlipayTradeRefundRequest`
- `AlipayTradeRefundModel`
- `AlipayTradeRefundResponse`

说明退款不是纯本地状态修改，而是真正接了支付宝退款接口。

### 6.6 mock 支付模式的作用

`AlipayConfig` 中有：

- `mockPayment`

这个开关的意义是：

- 开发联调时，如果不想真正走支付宝接口，可以走模拟流程
- 退款逻辑中也会判断该开关，决定是否真的调用支付宝退款 API

这对开发测试非常重要。

### 6.7 支付结果页怎么实现

用户端支付结果页位于：

- `frontend/user/src/views/payment/result.vue`

页面根据路由参数判断：

- 是否成功
- reservationId
- message

然后给用户提供：

- 查看订单详情
- 返回我的预约
- 重试支付
- 返回首页

所以支付模块实现是“后端负责支付与回调，前端负责承接与展示”。

## 7. AI 客服是怎么实现的

AI 客服相关代码主要在：

- 控制器：`src/main/java/com/murder/controller/AIController.java`
- 服务：`src/main/java/com/murder/service/impl/AIServiceImpl.java`
- 用户端 API：`frontend/user/src/api/ai.js`
- 用户端 AI 组件：`frontend/user/src/components/AICustomerServiceEnhanced.vue`
- 用户端工具：`frontend/user/src/utils/aiService.js`

### 7.1 AI 客服接口有哪些

`AIController` 当前提供：

- `POST /api/ai/chat`：AI 对话
- `GET /api/ai/recommend/{type}`：AI 推荐
- `POST /api/ai/log`：记录对话
- `GET /api/ai/faq`：获取常见问题
- `POST /api/ai/feedback`：提交反馈

### 7.2 前端 AI 客服怎么接入

用户端通过 `frontend/user/src/api/ai.js` 调用后端：

- `sendMessage(params)` -> `/ai/chat`
- `getAIRecommendation(type)` -> `/ai/recommend/{type}`
- `logConversation(params)` -> `/ai/log`
- `getFrequentQuestions()` -> `/ai/faq`
- `submitFeedback(params)` -> `/ai/feedback`

真正的交互界面在：

- `frontend/user/src/components/AICustomerServiceEnhanced.vue`

这个组件实现了：

- 悬浮聊天入口
- AI 对话窗口
- 对话历史展示
- 未读消息提示
- 常见问题推荐
- AI 与人工客服切换

也就是说，它不是只做一个输入框，而是做成了一个完整的客服浮窗组件。

### 7.3 AI 服务端内部怎么处理

`AIServiceImpl` 中比较关键的实现点有：

#### 1）配置项读取
从配置中读取：

- `ai.provider`
- `ai.api-key`
- `ai.api-url`
- `ai.model`
- `temperature`
- `max-tokens`

说明 AI 层是可配置的，不是把模型参数硬编码在方法里。

#### 2）系统提示词构建
服务中有：

- `buildSystemPrompt(context)`

这个方法不是简单给模型一句“你是客服”，而是构建了比较长的系统提示词，强调：

- AI 是平台内部客服“小剧”
- 要基于本平台真实功能回答
- 不能胡乱承诺系统没有的功能

这说明项目用了“业务约束提示词”的方式来提升 AI 回答质量。

#### 3）上下文注入
AI 对话接口支持：

- `message`
- `sessionId`
- `history`
- `context`

所以它不是一次性问答，而是支持历史上下文的。

#### 4）模型调用方式
服务内部使用：

- `RestTemplate`

并根据不同 provider 构建请求体、请求头、解析响应。

当前项目配置实际使用的是：

- DeepSeek

从 `application.yml` 可见：

- `provider: deepseek`
- `model: deepseek-chat`

#### 5）兜底与扩展逻辑
除了实际模型调用外，`AIServiceImpl` 里还实现了：

- FAQ 返回
- 对话日志记录
- 反馈提交
- 推荐结果生成入口

虽然其中有些方法目前仍带 TODO 或较轻量实现，但接口和服务框架已经搭好了。

### 7.4 当前 AI 客服实现特点

这个项目的 AI 客服不是纯 RAG 系统，而是“提示词约束 + 平台知识 + 前端引导 + FAQ 兜底”的组合实现。

它的优点是：

- 接入成本低
- 体验明显
- 便于快速演示
- 可以在论文/答辩中体现 AI 创新点

## 8. AI 排班是怎么实现的

AI 排班相关代码主要在：

- 控制器：`src/main/java/com/murder/controller/AiScheduleController.java`
- 服务：`src/main/java/com/murder/service/impl/AiScheduleServiceImpl.java`
- 管理端 API：`frontend/admin/src/api/ai-schedule.js`

### 8.1 AI 排班做的事情是什么

它解决的是：

- 给门店未来一段时间内的剧本排期，自动推荐适合的 DM
- 辅助门店管理员做 DM 分配
- 生成排班摘要说明

也就是“AI 辅助排班”，而不是自动替代管理员完全决策。

### 8.2 排班接口

当前主要接口包括：

- `POST /api/ai/schedule/dm/suggest`：生成 DM 推荐方案
- `POST /api/ai/schedule/dm/confirm`：确认并应用分配方案
- `GET /api/ai/schedule/task/list`：查看 AI 排班任务历史

### 8.3 推荐方案如何生成

`AiScheduleServiceImpl.suggestDmAssignment(...)` 的核心流程大致是：

1. 确定门店范围与日期范围
2. 创建一条 AI 排班任务记录 `AiScheduleTask`
3. 查询该门店在时间区间内的排期 `ScriptSchedule`
4. 查询门店在职 DM 列表 `Dm`
5. 按规则过滤掉时间冲突 DM
6. 结合剧本标签、DM 风格、DM 评分、排班负载计算匹配分数
7. 为每个排期生成候选 DM 列表
8. 汇总生成分配建议与摘要说明
9. 更新任务结果

### 8.4 排班评分逻辑

实现上并不是直接把全部排班扔给大模型，而是先由后端规则引擎做基础评分：

- 剧本标签与 DM 风格匹配度
- DM 综合评分
- 当前已排班次数
- 时间冲突过滤

对应方法包括：

- `hasTimeConflict(...)`
- `calcScore(...)`
- `buildReason(...)`

这说明它的实现思路是：

**规则引擎做可控分配，AI 主要负责摘要解释与增强体验。**

### 8.5 AI 在排班中的作用

`callAiForDmSummary(...)` 会根据推荐结果生成总结说明。

也就是说，AI 排班模块中：

- 真正决定候选和匹配的是规则逻辑
- AI 负责对结果做自然语言总结

这种设计比“完全交给大模型决定排班”更稳定、更可控，也更适合实际业务。

### 8.6 已补上的安全限制

AI 排班最近已经补了两类关键限制：

1. AI 排班接口不再作为公开接口放行
2. 门店管理员只能操作自己门店的排班数据
3. 确认分配时会校验：
   - 排期是否属于当前门店
   - DM 是否属于该排期门店

这使 AI 排班从“能用”提升到了“基本安全可控”。

## 9. 人工客服与 WebSocket 是怎么实现的

相关代码主要在：

- `src/main/java/com/murder/controller/ServiceSessionController.java`
- `src/main/java/com/murder/service/impl/ServiceSessionServiceImpl.java`
- `src/main/java/com/murder/websocket/ServiceWebSocketHandler.java`
- 前端 AI 客服组件：`frontend/user/src/components/AICustomerServiceEnhanced.vue`

### 9.1 人工客服的业务流程

大致流程是：

1. 用户在 AI 客服窗口中切换到人工客服
2. 前端创建客服会话
3. 用户与管理员通过 WebSocket 实时收发消息
4. 管理端接入后进入人工客服模式
5. 对话结束后可以结束会话并做评价

### 9.2 WebSocket 实现方式

`ServiceWebSocketHandler` 维护了两类连接：

- 用户连接
- 管理员连接

它支持：

- 用户上线注册 session
- 管理员上线注册 session
- ping/pong 保活
- 指定用户推送消息
- 指定管理员推送消息
- 广播新会话通知给所有在线管理员

这说明人工客服模块不是轮询式聊天，而是真正做了实时通信。

## 10. 预约、支付、履约为什么是这个项目的主线

虽然项目功能很多，但从代码实现角度看，最核心的业务主线还是预约订单。

主要实体：

- `Reservation`

它串起了：

- 用户
- 门店
- 房间
- 剧本
- 排期
- DM
- 优惠券
- VIP 折扣
- 支付状态
- 退款状态
- 核销状态

因此可以把项目理解成：

**以 Reservation 为核心订单实体，向前连接用户消费流程，向后连接门店履约流程。**

支付、退款、核销、评价、通知，其实都围绕这条线展开。

---

## 11. 这个项目的实现亮点

### 11.1 双前端 + 单体后端组合清晰

用户端与管理端职责分离明显，代码边界清楚，适合毕业设计展示。

### 11.2 AI 功能不是装饰，而是嵌入到业务中

项目中的 AI 不是只放一个“聊天框”，而是做了两条落地线：

- AI 客服
- AI 排班

这两个点都能作为项目创新点来讲。

### 11.3 支付流程接近真实业务系统

支持：

- 创建支付单
- 支付宝沙箱
- 异步通知
- 同步回跳
- 退款处理
- 支付结果页

这比普通课程设计只做“支付成功假页面”更完整。

### 11.4 权限模型比较细

不仅有角色区分，还有员工权限码，体现出项目对门店协同的真实考虑。

## 12. 当前实现中需要注意的地方

### 12.1 配置项安全性

当前开发配置中包含 AI 与支付宝相关参数。实际部署时建议：

- 不要把真实密钥直接写进仓库
- 改成环境变量或外部配置中心

### 12.2 AI 客服与 AI 排班不要混为一谈

这两个模块虽然都叫 AI，但实现方式不同：

- AI 客服：偏对话与回答
- AI 排班：偏规则引擎 + AI 摘要

答辩时必须讲清楚，否则老师容易追问“你是不是都交给大模型处理了”。

### 12.3 支付结果以异步通知为准

前端结果页只是体验层反馈，真正的订单支付状态仍应以后端收到支付宝通知后的更新为准。

## 13. 如果在答辩中讲“具体怎么实现”可以怎么说

可以用下面这个简化版本：

> 本项目后端采用 Spring Boot 单体架构，使用 MyBatis-Plus 操作 MySQL，通过 JWT 实现用户端和管理端鉴权。前端分为用户端和管理端，两套系统都使用 Vue3 + Vite 开发，通过 Axios 调用统一的 `/api` 接口。
>
> 在业务实现上，我把预约订单作为核心主线。用户先浏览剧本和门店，选择排期创建预约，再进入支付模块。支付模块通过后端接入支付宝沙箱，由后端生成支付表单并处理异步通知与同步回跳，支付成功后更新预约状态。退款时也由后端调用支付宝退款接口并更新订单状态。
>
> AI 部分我做了两个方向。第一个是 AI 客服，通过后端封装 DeepSeek 对话接口，结合系统提示词和上下文实现平台专属客服回答；第二个是 AI 排班，通过规则引擎先计算 DM 与排期的匹配度，再由 AI 对分配结果生成摘要，提高门店排班效率。
>
> 此外，系统还通过 WebSocket 实现人工客服和通知推送，使用户端和管理端都能获得更实时的交互体验。
