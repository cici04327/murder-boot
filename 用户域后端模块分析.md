# 用户域后端模块详细分析

## 1. 认证方式

### 1.1 JWT令牌认证
- **密钥管理**：双密钥系统
  - `userSecretKey`：普通用户登录
  - `adminSecretKey`：管理员登录（支持长期登录，忽略过期）

- **令牌构成（Claims）**
  - `userId`：用户ID
  - `username`：用户名
  - `phone`：手机号
  - `role`：用户角色（USER/SUPER_ADMIN/STORE_ADMIN）
  - `storeId`：门店ID（仅限STORE_ADMIN）

- **验证流程**
  1. JwtTokenInterceptor拦截请求，提取Authorization头
  2. 根据请求URI或X-Client-Type头智能选择密钥
  3. 调用JwtUtil.parseJWT()验证签名和有效期
  4. 解析Claims存入BaseContext，供业务层使用

- **关键特性**
  - `/api/admin`开头自动用adminSecretKey
  - 管理员操作（预约确认、检查入住等）忽略过期
  - 用户端严格检查过期时间

### 1.2 用户角色体系
```
USER           → 普通用户（userSecretKey）
SUPER_ADMIN    → 总部管理员（adminSecretKey，无storeId限制）
STORE_ADMIN    → 门店管理员（adminSecretKey，必须绑定storeId）
ADMIN          → 废弃角色（不允许在用户端登录）
```

### 1.3 登录日志记录
- 记录内容：userId、IP、地理位置、设备信息、登录时间、状态
- 表结构：`user_login_log`
- 更新User表的lastLoginTime字段

- 条件：确认文本="确认注销"且密码正确
- 操作：
  - status=0（禁用）
  - phone/email清空
  - 所有验证状态重置
  - isDeleted=1（逻辑删除）

## 3. 积分与优惠券

### 3.1 积分系统（Points）

**积分类型和来源**
| 来源代码 | 来源名称 | 描述 |
|---------|---------|------|
| 1 | 注册赠送 | 新用户注册 |
| 2 | 消费获得 | 预约支付 |
| 3 | 签到奖励 | 每日签到 |
| 4 | 活动奖励 | 活动参与 |
| 5 | 管理员调整 | 后台调整 |

**积分操作**

1. **增加积分** - `addPoints(userId, points, reason)`
   - VIP加成机制：获取用户VIP倍率，应用到积分
   - 公式：最终积分 = 原始积分 × VIP倍率
   - 记录UserPointsRecord，type=1（增加）
   - 更新User.points

2. **扣减积分** - `deductPoints(userId, points, reason)`
   - 检查积分是否足够
   - 扣减后newPoints < 0则抛异常
   - 记录UserPointsRecord，type=2（扣减）

3. **每日签到** - `signIn(userId)`
   - 检查当天是否已签到
   - 奖励10积分
   - 签到失败：同一天重复签到

4. **积分兑换优惠券** - `exchangeCoupon(userId, couponId)`
   - 扣减用户积分（coupon.exchangePoints）
   - 创建UserCoupon记录
   - 记录积分变动
   - 当日限制：同一优惠券只能兑换1次

### 3.2 积分记录（UserPointsRecord）

| 字段 | 类型 | 说明 |
|------|------|------|
| userId | Long | 用户ID |
| points | Integer | 积分数（正数增加，负数扣减） |
| type | Integer | 类型（1增加2扣减） |
| source | Integer | 来源（见上表） |
| description | String | 描述（含VIP加成信息） |

### 3.3 优惠券系统（Coupon）

**优惠券类型**
| 类型代码 | 类型名称 | 说明 |
|---------|---------|------|
| 1 | 满减券 | discountValue为减免金额 |
| 2 | 折扣券 | discountValue为折扣比例（0.8=8折） |
| 3 | 代金券 | discountValue为代金额 |

**优惠券属性**

| 字段 | 类型 | 说明 |
|------|------|------|
| name | String | 优惠券名称 |
| type | Integer | 优惠券类型 |
| discountValue | BigDecimal | 折扣值 |
| minAmount | BigDecimal | 最低消费金额 |
| totalCount | Integer | 发行总量 |
| remainCount | Integer | 剩余数量（已领取数） |
| validStartTime | LocalDateTime | 有效期开始 |
| validEndTime | LocalDateTime | 有效期结束 |
| exchangePoints | Integer | 兑换所需积分（0=免费领取） |
| status | Integer | 状态（1上架0下架） |

**优惠券生命周期**

1. **发行阶段**
   - 创建Coupon记录
   - remainCount=totalCount
   - status=1（上架）

2. **领取阶段** - `receiveCoupon(userId, couponId)`
   - 检查优惠券状态和有效期
   - 检查是否已领取（未使用或已使用的都算已领取）
   - 检查剩余数量
   - 创建UserCoupon记录，status=1（未使用）
   - 扣减remainCount

3. **使用阶段**
   - 在预约支付时选择使用的优惠券
   - 更新UserCoupon，status=2（已使用），useTime=now，orderId=订单ID
   - 计算优惠金额应用到订单

4. **过期阶段** - `getUserCoupons()`自动更新
   - 查询时自动更新：status=1且expireTime < now → status=3（已过期）
   - 已使用的优惠券保持status=2

### 3.4 用户优惠券（UserCoupon）

| 字段 | 类型 | 说明 |
|------|------|------|
| userId | Long | 用户ID |
| couponId | Long | 优惠券ID |
| status | Integer | 状态（1未使用2已使用3已过期） |
| orderId | Long | 关联订单ID（已使用时填写） |
| receiveTime | LocalDateTime | 领取时间 |
| useTime | LocalDateTime | 使用时间 |
| expireTime | LocalDateTime | 过期时间 |

**查询功能**
- `getUserCoupons(userId, page, pageSize, status)`：分页查询用户优惠券
- `getAvailableCoupons(userId, orderAmount)`：查询可用优惠券（需满足最低消费金额）

| 字段 | 类型 | 说明 |
|------|------|------|
| name | String | 套餐名称 |
| level | Integer | 会员等级（1普通2银卡3金卡4钻石） |
| durationType | Integer | 时长类型（1月2季3半年4年） |
| durationDays | Integer | 时长天数 |
| originalPrice | BigDecimal | 原价 |
| currentPrice | BigDecimal | 现价 |
| discountRate | BigDecimal | 折扣率（0.8=8折） |
| pointMultiplier | BigDecimal | 积分倍率 |
| couponCount | Integer | 每月赠送优惠券数 |
| priorityBooking | Integer | 预约优先（1有0无） |
| exclusiveService | Integer | 专属客服（1有0无） |
| birthdayGift | Integer | 生日特权（1有0无） |
| exclusiveBadge | Integer | 专属徽章（1有0无） |
| specialDiscount | BigDecimal | 专属折扣（0.95=95折） |
| features | String | 权益列表JSON |
| tag | String | 标签（推荐、最划算等） |
| sortOrder | Integer | 排序 |
| status | Integer | 状态（1上架0下架） |

### 4.2 VIP购买流程

**购买阶段**
1. 用户选择套餐和支付方式
2. 调用`purchaseVip(userId, packageId, paymentMethod)`
3. 当前仅支持支付宝（ALIPAY）
4. 创建UserVip记录，status=0（待支付）
5. 生成支付宝支付链接并返回

**支付宝回调处理** - `handleAlipayNotify(params)`
1. 验证支付宝签名（RSA验证）
2. 检查回调appId、订单号、金额是否匹配
3. 若trade_status=TRADE_SUCCESS/TRADE_FINISHED则激活VIP
4. 返回"success"给支付宝

**支付宝同步回跳** - `handleAlipayReturn(params)`
1. 验证签名和订单信息
2. 若支付成功则激活VIP
3. 重定向到前端VIP结果页面

### 4.3 VIP激活逻辑

**多次续费的时间叠加机制**
```
激活逻辑：
1. 查询getLatestPaidVip(userId) → 获取已激活的endTime最晚的VIP
2. 若latestPaidVip.endTime > now：
   startTime = latestPaidVip.endTime（续费从上一个结束时间开始）
   else：
   startTime = now（首次或已过期，从现在开始）
3. endTime = startTime + durationDays
```

**激活时的额外操作**
- 刷新User表的vipLevel和vipExpireTime
- 若startTime <= now（立即生效），发放当月优惠券

### 4.4 用户VIP记录（UserVip）

| 字段 | 类型 | 说明 |
|------|------|------|
| userId | Long | 用户ID |
| packageId | Long | 套餐ID |
| level | Integer | VIP等级 |
| startTime | LocalDateTime | 开始时间（激活时确定） |
| endTime | LocalDateTime | 结束时间 |
| originalPrice | BigDecimal | 原价 |
| actualPrice | BigDecimal | 实付金额 |
| paymentMethod | String | 支付方式（ALIPAY/WECHAT/POINTS） |
| orderNo | String | 订单号（格式：USER_{userId}_{timestamp}） |
| transactionId | String | 第三方交易号 |
| status | Integer | 状态（0待支付1生效中2已过期3已取消） |
| autoRenew | Integer | 自动续费（0否1是） |

### 4.5 VIP权益查询 - `getUserVipInfo(userId)`

返回信息：
```json
{
  "hasVip": boolean,          // 是否有有效VIP
  "vipLevel": integer,        // VIP等级
  "currentVip": {
    "level": integer,
    "startTime": datetime,
    "endTime": datetime,
    "packageName": string
  },
  "nextVip": {                // 近期待激活的VIP
    "level": integer,
    "startTime": datetime,
    "endTime": datetime
  },
  "latestVip": {              // 最新的任何VIP（已激活/待激活）
    "endTime": datetime
  }
}
```

### 4.6 VIP核心查询方法

1. **getCurrentVip(userId)**
   - 条件：status=1 AND startTime <= now AND endTime > now
   - 返回：当前生效的VIP

2. **getLatestPaidVip(userId)**
   - 条件：status=1（已激活）
   - 排序：endTime DESC，取第一条
   - 用途：续费时计算开始时间

3. **getPointMultiplier(userId)**
   - 查询currentVip或latestVip
   - 返回VipPackage的pointMultiplier
   - 若无VIP返回1.0

## 5. 关键数据关系与潜在注意点

### 5.1 核心数据依赖关系图

```
User（用户核心）
├── points → UserPointsRecord（积分记录）
├── vipLevel, vipExpireTime ← UserVip（VIP记录）
│                             └── packageId → VipPackage（套餐配置）
├── UserAddress（收货地址）
├── UserSettings（用户设置）
├── UserPreference（用户偏好）
├── UserCoupon（用户优惠券）
│   └── couponId → Coupon（优惠券配置）
└── UserLoginLog（登录日志）
```

### 5.2 关键注意点

#### 5.2.1 VIP积分加成问题
- **当前实现**：每次加积分时查询VIP倍率，动态计算
- **问题**：若VIP在加积分和扣积分之间变化，可能导致不一致
- **建议**：考虑在addPoints记录原始积分和倍率，便于对账

#### 5.2.2 积分兑换优惠券的并发问题
- **当前机制**：当日同一优惠券仅能兑换1次
- **实现细节**：交易内扣积分→创建UserCoupon→记录变动
- **风险**：未加分布式锁，高并发可能导致扣减多次但只创建1条优惠券

#### 5.2.3 VIP续费时间叠加
- **设计**：多次续费时间叠加（调用getLatestPaidVip查询最晚endTime）
- **陷阱**：getCurrentVip只查startTime <= now的VIP，若有未来生效的续费订单会被忽略
- **已修复**：activateVipOrder使用getLatestPaidVip避免此问题

#### 5.2.4 优惠券过期自动更新
- **机制**：查询用户优惠券时自动更新status=1 AND expireTime < now → 3
- **问题**：
  1. 若某个用户长期不查优惠券，过期数据持续存储
  2. 每次查询都要扫表更新，高并发性能差
  3. 建议添加定时任务批量更新过期优惠券

#### 5.2.5 用户缓存策略
- **缓存键**：`user:{userId}`
- **缓存操作**：
  - getUserById时缓存用户信息
  - updatePassword、账户注销时清除缓存
- **问题**：
  1. VIP激活时更新user的vipLevel/vipExpireTime，未清除缓存
  2. 用户自行修改昵称等，可能导致缓存不一致
  3. 建议在更新用户信息后主动清除缓存

#### 5.2.6 支付安全性
- **支付宝验签**：使用RSA验签检查回调数据真实性 ✓
- **金额校验**：检查回调金额与订单金额是否一致 ✓
- **订单幂等性**：status=0时才激活，重复回调不会重复激活 ✓
- **缺陷**：未检查回调是否重复（可能被攻击者多次调用notify）
  - 建议：记录已处理过的trade_no，重复回调直接返回success

#### 5.2.7 用户验证状态管理
- **手机/邮箱验证**：通过短信/邮件验证码
- **实名认证**：直接保存身份证号（未做格式/真实性验证）
- **问题**：
  1. 账户注销时phone/email被清空，但phoneVerified/emailVerified状态未清
  2. 建议：注销时将这些字段同步重置为0

#### 5.2.8 用户注销的完整性
- **当前操作**：
  - status=0、isDeleted=1、phone/email清空
  - 验证状态重置
- **未处理**：
  1. 用户的预约记录、优惠券、VIP权益如何处理？
  2. 用户的积分记录是否保留？
  3. 建议：明确定义逻辑删除后的数据可见性和生命周期

#### 5.2.9 用户角色和权限细节
- **登录区分**：
  - login()：仅允许role=USER的用户
  - adminLogin()：仅允许SUPER_ADMIN或STORE_ADMIN
- **问题**：
  1. 没有基于角色的权限细粒度控制（如STORE_ADMIN仅能管理自己的门店）
  2. 权限检查散落在各个Controller中
  3. 建议：统一权限检查框架（如@PreAuthorize或自定义注解）

#### 5.2.10 JWT长期有效性问题
- **管理员令牌**：ignoreExpiration=true，长期有效
- **问题**：
  1. 管理员登出后仍可用旧令牌进行操作
  2. 风险：员工离职后其令牌仍有效
  3. 建议：实现令牌黑名单或logout操作

#### 5.2.11 用户偏好学习机制
- **记录**：UserPreference记录用户对category/difficulty/tag的偏好
- **问题**：
  1. 如何触发更新？在浏览、收藏、预约时自动累计？
  2. 当前代码未见实际更新逻辑
  3. 建议：明确定义触发时机和加权公式

#### 5.2.12 用户设置JSON序列化
- **存储**：privacySettings/notificationSettings存储为JSON字符串
- **问题**：
  1. 序列化/反序列化逻辑？
  2. 字段变更时的版本兼容性？
  3. 建议：定义清晰的JSON schema

### 5.3 交叉模块影响

| 场景 | 涉及模块 | 关键点 |
|------|---------|--------|
| 预约支付 | Reservation + Coupon + UserPointsService | 应用优惠券减价，消费获得积分 |
| VIP激活 | UserVip + VipPackage + Coupon + UserPointsService | 发放优惠券，积分倍率改变 |
| 用户注销 | User + UserCoupon + UserVip + UserPointsRecord | 级联删除或标记的策略？ |
| 积分兑换 | UserPointsService + CouponService | 原子性：同时扣积分和发券 |

## 总结

用户域后端采用**JWT + Role-Based**认证架构，围绕**User、VIP、Points、Coupon**四个核心概念构建。

**关键特性**：
1. ✓ 双密钥JWT系统，支持长期管理员登录
2. ✓ VIP时间叠加机制，支持续费
3. ✓ 积分动态加成，支持VIP倍率
4. ✓ 优惠券灵活配置，支持积分兑换
5. ✓ 支付宝集成完整，验签和金额校验
