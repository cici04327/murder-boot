## 1. 项目概述

### 1.1 项目名称
剧本杀预约与门店管理系统

### 1.2 项目定位
本项目是一个围绕剧本杀门店经营场景构建的综合服务平台，不是单纯的信息展示网站，也不是传统电商系统。它解决的是剧本杀行业中“线上展示 + 在线预约 + 线下履约 + 门店协同 + 会员营销 + 智能服务”的完整闭环问题。

系统的核心目标主要包括四点。第一，为普通用户提供剧本浏览、门店查询、在线预约、订单管理、VIP 会员、优惠券、拼团、评价反馈、推荐与客服服务。第二，为门店提供预约处理、DM 分配、员工协同、经营统计和门店运营能力。第三，为平台管理员提供用户、门店、剧本、通知、反馈和统计等全局管理能力。第四，通过 AI 客服和人工客服机制提升咨询体验。

### 1.3 这个项目要解决的真实问题
这个项目要解决的并不是抽象的软件开发问题，而是剧本杀行业中的实际业务痛点。很多门店仍依赖微信、电话或人工登记完成预约，这会导致预约流程低效、门店与剧本信息分散、退款改期不规范、DM 分配不清晰、通知不及时以及统计分析能力不足。随着会员营销、优惠券活动、拼团组局、个性化推荐和智能客服越来越重要，传统人工管理方式已经难以支撑完整运营。因此，本项目本质上是一个面向剧本杀门店数字化运营的综合平台。

## 2. 技术架构与技术栈

### 2.1 整体架构
本项目采用的是 Spring Boot 单体后端 + Vue 双前端 的前后端分离架构。

后端使用 Spring Boot 对外提供 RESTful API。前端分为两个项目：一个是用户端，主要面向普通用户；另一个是管理端，主要面向系统管理员、门店管理员和门店员工。数据库使用 MySQL 存储核心业务数据，Redis 用于缓存和辅助性能优化。系统还接入了 WebSocket 处理实时通知和人工客服消息，接入支付宝完成支付流程，并通过 DeepSeek 官方 API 提供 AI 客服能力。

### 2.2 后端技术栈
- Spring Boot：主框架，负责接口、事务、配置、依赖注入
- MyBatis-Plus：简化 CRUD、分页与条件查询
- MySQL：存储业务核心数据
- Redis：缓存与辅助状态存储
- JWT：用户端和管理端鉴权
- Knife4j / OpenAPI：接口文档
- WebSocket：通知与客服实时消息
- 支付宝：预约和 VIP 支付
- DeepSeek：AI 客服能力

### 2.3 前端技术栈
用户端使用 Vue 3、Vue Router、Pinia、Axios、Element Plus、ECharts、Vite。管理端同样使用 Vue 3 技术栈，并增加 xlsx 和 file-saver 以支持导出能力。

### 2.4 开发端口与联调方式
从 Vite 配置可以确认，用户端开发端口是 3001，管理端开发端口是 3000，后端端口是 8080。两个前端都通过代理把 `/api` 和 `/upload` 请求转发给 Spring Boot，因此开发阶段不需要额外处理跨域问题。

## 4. 角色体系与权限模型

### 4.1 普通用户
普通用户通过用户端完成剧本浏览、门店查询、在线预约、订单管理、VIP 购买与续费、优惠券与积分使用、拼团、评价反馈、收藏、浏览历史、通知查看、AI 客服和人工客服等操作。

### 4.2 系统管理员（SUPER_ADMIN）
系统管理员是平台级后台角色，主要负责用户管理、门店管理、剧本管理、员工管理、优惠券管理、VIP 套餐管理、留言反馈、通知公告和平台统计等工作。

### 4.3 门店管理员（STORE_ADMIN）
门店管理员是门店级后台角色，主要负责本门店信息维护、经营看板、DM 管理、员工管理、预约处理、排期管理和营收日报查看等工作。

### 4.4 门店员工（STORE_STAFF）
门店员工并不是单一权限角色，而是根据 `StoreEmployee` 中的岗位和后台角色进一步细分为三类：MANAGER、DM、CLERK。

在项目实现中，position 1 和 2 对应 MANAGER，position 3 对应 DM，其他岗位默认归为 CLERK。这一设计使系统里的后台员工角色与现实门店岗位相对应。

### 4.5 员工权限码模型
`StoreEmployeeServiceImpl` 中定义了默认权限码：

- MANAGER：`reservation:view,reservation:checkin,reservation:complete,reservation:assign_dm,refund:process,employee:manage,report:view,notification:view`
- DM：`reservation:view,reservation:complete,notification:view`
- CLERK：`reservation:view,reservation:checkin,notification:view`

因此，这个项目的权限体系不是“只有 role”，而是“登录角色 + staffRole + permissionCodes”三层联合判断。

## 5. 核心业务模块

### 5.1 用户与账号模块
用户相关接口主要在 `UserController` 中。系统支持用户注册、登录、资料修改、头像上传、密码修改、手机号/邮箱绑定、实名认证、登录日志、隐私设置、通知设置、偏好设置和账号注销。

用户端登录成功后，`userStore` 会把 token 和 userInfo 保存到 `localStorage`。其中 token 存在 `token` 字段，用户信息存在 `userInfo` 字段。

### 5.2 剧本模块
剧本实体 `Script` 中定义了名称、分类、封面、图片、描述、类型、难度、人数、时长、价格、标签、评分、是否独家、上下架状态等字段。对用户来说，剧本模块承担内容展示和预约决策入口的作用；对后台来说，剧本模块承担剧本新增、分类管理、排期管理、角色管理、评价查看等作用。

### 5.3 门店模块
门店实体 `Store` 中定义了门店名称、地址、电话、图片、营业时间、经纬度、评分、状态以及门店登录账号密码。这意味着门店在系统中不只是一个展示对象，还具备门店后台登录能力。

### 5.4 预约模块
预约模块是整个系统的业务主线。`Reservation` 实体包含订单号、用户 ID、门店 ID、房间 ID、剧本 ID、拼团 ID、排期 ID、DM ID、人数、预约时间、总价、优惠金额、VIP 折扣金额、实付金额、状态、支付状态、退款状态、核销码、核销状态等关键字段。

`ReservationServiceImpl.create()` 是这个项目最值得深入掌握的方法之一，因为这里完成了完整的预约创建逻辑：先校验排期是否存在、是否已开始、是否关闭、是否满员，再校验房间时间冲突，然后由服务端重算总价，自动识别 VIP 折扣，尝试使用优惠券，写入订单，更新排期人数，必要时自动创建拼团，并向用户和后台发送通知。

这个流程说明系统不是“前端算好价格，后端只存一下”，而是由后端主导关键业务校验和金额计算，保证数据安全和流程一致性。

### 5.5 支付模块
本项目当前实际只支持支付宝支付，适用场景包括预约支付和 VIP 支付。相关回调接口在 `WebMvcConfig` 中被设置为公开访问，因为支付宝异步通知必须绕过登录鉴权。

预约支付成功后，系统会更新支付状态、支付时间和核销码，并发送支付成功通知。VIP 支付成功后，则会进入 VIP 激活逻辑。

### 5.6 VIP 模块
VIP 相关接口在 `VipController` 中，包括套餐列表、套餐详情、购买、续费、用户 VIP 信息、历史记录、月度体验券发放状态和状态检查。

VIP 相关的核心实体有两个：`VipPackage` 和 `UserVip`。前者负责定义 VIP 套餐配置，后者记录用户历史会员订单。用户当前的 `vipLevel` 和 `vipExpireTime` 冗余存储在 `User` 实体中，方便前端快速读取和业务判断。

这个项目中，VIP 不只是一个“是否会员”的布尔标记，而是完整的会员业务体系，和折扣、积分倍率、月度体验券、生日礼券、优先预约等能力联动。

### 5.7 优惠券与积分模块
优惠券使用 `Coupon` 和 `UserCoupon` 两类数据组织。积分通过 `UserPointsRecord` 记录。系统支持平台发券、VIP 发券、积分兑换优惠券、下单用券、取消预约时回退优惠券等能力。

当前项目内已经统一过积分与优惠券规则，AI 客服和前端本地知识库均按以下规则对齐：每日签到 +10，完成预约游戏 +100，收藏达到里程碑 +20，VIP 有积分倍率加成，积分可用于兑换优惠券。

### 5.8 拼团模块
拼团相关实体是 `GroupOrder` 和 `GroupMember`，相关接口在 `GroupOrderController` 中。系统支持拼团大厅、热门拼团、拼团详情、创建、加入、退出、取消和我的拼团。

更重要的是，这个项目不是把拼团做成完全独立的模块，而是和预约流程联动。在 `ReservationServiceImpl.create()` 中，如果用户预约人数小于剧本要求人数，系统会自动创建或挂接一个拼团房间。

### 5.9 推荐模块
项目包含推荐能力，接口在 `RecommendationController` 中，包含个性化推荐、热门推荐、新品推荐、协同过滤、相似推荐和基于历史推荐等能力。推荐模块可以结合浏览历史、收藏行为和热门排行等数据进行推送。

### 5.10 通知模块
系统包含用户通知和管理端通知两套通知体系。用户端可查看预约成功通知、支付成功通知、VIP 月度券到账通知、系统公告等。管理端则有通知中心页面。由于项目引入了 WebSocket，因此通知具备实时推送能力，不是完全依赖轮询。

### 5.11 AI 客服与人工客服模块
这是项目最有辨识度的创新模块之一。AI 客服相关接口位于 `AIController`，包括 `/api/ai/chat`、`/api/ai/recommend/{type}`、`/api/ai/log`、`/api/ai/faq` 和 `/api/ai/feedback`。

当前 AI 客服采用的是“DeepSeek 大模型 + 前端本地知识库兜底”的混合模式。前端组件 `AICustomerService.vue` 会先把用户消息、最近 10 轮对话历史、当前页面路径以及当前用户的昵称和 VIP 等级一起发给后端。后端 `AIServiceImpl` 中的 `chatWithDeepSeek()` 方法会调用 DeepSeek 官方 `chat/completions` 接口，而 `buildSystemPrompt()` 则把本平台真实实现的业务规则写入系统提示词中，包括角色体系、预约流程、退款规则、VIP 规则、积分规则、优惠券规则、拼团规则、支付方式、通知方式以及人工客服方式等。也就是说，当前 AI 已经不是一个通用聊天机器人，而是一个“项目熟悉版 AI 客服”。

如果 DeepSeek 调用失败，前端还会回退到本地 `knowledgeBase`，继续通过关键词匹配返回固定答案，从而保证用户不会直接看到错误提示。此外，系统还提供人工客服能力。用户可以发起人工客服会话，接口位于 `ServiceSessionController`，支持创建会话、查询活跃会话、发送消息、获取消息列表、关闭会话和评分。管理端也支持管理员接入、发送消息和关闭会话。这意味着客服模块的完整链路是“AI 先接待，复杂问题可转人工，人工通过 WebSocket 实时处理”。

## 7. 关键数据实体与关系

### 7.1 核心实体
从 `entity` 目录可以看出，这个项目已经建立了比较完整的业务实体体系。核心实体包括：`User`、`Store`、`StoreEmployee`、`Script`、`ScriptCategory`、`ScriptRole`、`ScriptSchedule`、`Reservation`、`VipPackage`、`UserVip`、`Coupon`、`UserCoupon`、`GroupOrder`、`GroupMember`、`ServiceSession`、`ServiceMessage`、`Feedback`、`UserNotification`、`AdminNotification`、`UserPointsRecord`、`Review`、`ScriptReview`、`StoreReview` 等。

### 7.2 关键关系
一个用户可以有多条预约记录，一个门店可以有多个员工，一个剧本可以有多个排期，一个预约会关联用户、门店、剧本、房间、排期和可能的拼团记录。用户的历史 VIP 记录保存在 `UserVip` 中，而当前会员等级冗余保存在 `User` 表的 `vipLevel` 中。一个用户可以持有多张 `UserCoupon`，一个拼团 `GroupOrder` 可以拥有多个 `GroupMember`，一个客服会话 `ServiceSession` 可以对应多条 `ServiceMessage`。从这些关系可以看出，系统数据库设计不是简单的用户表和订单表，而是围绕剧本杀门店业务做了完整建模。

### 7.3 为什么这样设计
这种实体拆分方式的好处在于：一方面可以让业务含义更清晰，例如把“套餐定义”和“用户会员记录”拆开，把“优惠券模板”和“用户实际领取券”拆开；另一方面有利于后续扩展，例如以后增加更多会员权益、更多通知场景、更多推荐算法，都可以在现有实体基础上扩展，而不需要推翻已有模型。

## 8. 鉴权、安全与上下文设计

### 8.1 JWT 鉴权
项目使用 JWT 作为统一登录态方案，并且区分用户端和管理端密钥。配置类 `JwtProperties` 中定义了 `userSecretKey`、`userTtl`、`adminSecretKey`、`adminTtl` 等字段，说明系统对用户端和管理端的 token 生命周期与密钥进行了分离设计。

### 8.2 统一拦截器
后端统一通过 `JwtTokenInterceptor` 对 `/api/**` 请求进行拦截，再在 `WebMvcConfig` 中配置白名单接口。公开接口包括：用户注册登录、管理端登录、支付回调、部分公开剧本/门店/文章/推荐/统计接口、AI 客服接口、反馈提交以及接口文档等。

`JwtTokenInterceptor` 在 token 校验通过后，会把用户 ID、role、storeId、employeeId、staffRole、permissionCodes、dmId 等信息写入 `BaseContext`。这意味着后端各层业务逻辑可以直接基于当前线程上下文判断操作者身份，而不需要在每个接口层层传递这些参数。

### 8.3 安全设计亮点
这个项目的安全设计亮点主要体现在三点。第一，权限不是只靠前端隐藏菜单，而是后端也严格校验。第二，员工端权限不是笼统的“员工可操作”，而是用 permission code 做细粒度控制。第三，预约相关的敏感操作带有业务范围限制，例如 DM 只能查看和操作分配给自己的预约，门店管理员只能操作自己门店的数据，普通用户只能访问自己的订单和资料。

同时，用户端登录请求封装还内置了异常处理、401 统一退出、网络错误重试等逻辑，进一步提升了交互安全性与稳定性。

## 9. AI 客服与人工客服（单独展开）

### 9.1 DeepSeek 接入方式
当前后端 `application.yml` 中的 AI provider 已经配置为 `deepseek`，API 地址是 DeepSeek 官方 `chat/completions` 接口。`AIServiceImpl` 中已经实现了 `chatWithDeepSeek()`，因此现在不是 mock 模式，而是真实调用 DeepSeek 官方接口。

### 9.2 AI 为什么能“懂项目”
因为 `buildSystemPrompt()` 已经不是通用客服模板，而是项目熟悉版提示词。这里面写死了平台角色体系、预约主流程、退款规则、改期规则、VIP 体系、积分规则、优惠券规则、拼团规则、支付方式、通知方式和人工客服方式。并且前端会把当前用户的 `vipLevel` 与当前页面一起传给后端，所以 AI 不只是知道“用户问了什么”，还知道“用户现在在哪个页面、是不是 VIP、VIP 是几级”。

### 9.3 本地兜底知识库
前端 `AICustomerService.vue` 中维护了一份本地关键词知识库，它在 AI 服务失败时起作用。当前已经与后端规则对齐的内容包括支付方式、退款规则、积分规则、VIP 等级权益以及退款到账方式等。这样就算大模型暂时不可用，用户也能得到基于项目真实规则的兜底回答。

### 9.4 VIP 状态与 AI 联动
项目已经额外做了两层优化：用户端 layout 会在挂载和登录状态变化时全局刷新 VIP 状态，VIP 页面也会在获取最新 VIP 信息后写回 `userStore`。这样 AI 客服在读取 `vipLevel` 时拿到的是最新状态，而不是旧缓存。

## 12. 后续优化方向

### 12.1 AI 客服升级为知识库检索模式
当前系统中的 AI 客服已经接入了 DeepSeek，并结合后端 `buildSystemPrompt()` 与前端本地 `knowledgeBase` 形成了一套“提示词 + 兜底知识库”的实现方式。它的优点是实现成本较低、接入速度快、效果直观，能够较好地回答预约流程、退款规则、VIP 权益、积分规则、优惠券说明、支付方式、通知方式和人工客服等问题。

但从长期维护角度来看，这种方式仍然存在明显局限。第一，知识主要写在后端提示词和前端本地规则中，维护时需要同时修改多处代码，容易出现规则不一致的问题。第二，随着业务功能不断增多，例如门店活动、更多 VIP 策略、更多客服问答场景，这种硬编码模式会逐渐降低维护效率。第三，AI 回答虽然已经比较贴近平台，但本质上仍然属于“提示词驱动”，如果后续要做更强的知识管理和内容迭代，仍需要更结构化的知识组织方式。

因此，后续更合理的优化方向，是将 AI 客服从当前的提示词模式升级为真正的知识库检索模式。具体来说，可以把预约规则、退款规则、VIP 权益、优惠券说明、门店服务信息、常见问题和运营说明等内容沉淀为统一的结构化知识数据，例如知识库表、配置文件或独立知识中心。用户提问时，系统先根据问题、当前页面和用户身份信息检索最相关的知识片段，再将这些检索结果与用户问题一起发送给大模型，由大模型基于检索结果生成回答。这样既能保留大模型自然表达的优势，又能提高回答的准确性、一致性和可维护性。

### 12.2 推荐系统结合用户画像做精细化推荐
当前系统已经实现了推荐模块，支持个性化推荐、热门推荐、新品推荐、协同过滤推荐、相似推荐以及基于历史记录的推荐能力。现有实现已经能够基于用户浏览、收藏和部分行为数据完成基础推荐，这使系统具备了从静态展示向智能推荐发展的基础条件。

但从推荐效果角度来看，当前推荐逻辑仍然有进一步优化空间。剧本杀用户在实际消费中存在明显的个体差异，例如有人偏好情感本，有人偏好推理本；有人喜欢 4 到 6 人的小规模本，有人更偏好多人的社交场景；有人对价格比较敏感，有人更看重门店口碑、时长或难度。如果后续只依赖较少的行为信号，推荐结果虽然能做到“有一定相关性”，但还不能完全做到“贴合不同用户真实偏好”。

因此，推荐系统的下一步优化方向，是进一步结合用户画像、浏览历史、收藏行为、预约记录、评价信息和消费偏好等数据，对用户兴趣进行更细粒度的建模。具体可以将推荐过程拆分为规则过滤、用户画像建模和推荐排序三个层次：首先通过规则过滤剔除不适合当前用户场景的剧本，其次根据历史行为构建用户兴趣标签和偏好权重，最后结合协同过滤、相似推荐和热门推荐结果进行综合排序。通过这种方式，可以让推荐结果不仅考虑“哪些剧本热门”，还考虑“哪些剧本更适合这个用户”，从而提升推荐命中率、点击率和预约转化率。

## 13. 系统链路总图（文字版）

### 13.1 用户端请求链路
用户端访问系统时，浏览器首先加载 Vue 页面。页面中的接口请求统一走 `frontend/user/src/utils/request.js`，请求会自动从 `userStore` 中读取 token，并把 token 放在请求头中发送给后端。请求到达 Spring Boot 后，会先经过 `JwtTokenInterceptor` 判断该接口是否需要登录、是否需要特定角色、token 是否有效。通过校验后，拦截器会把当前用户的身份信息写入 `BaseContext`，后续 Controller 和 Service 就可以直接读取当前用户、当前角色、当前门店和当前权限码等信息。然后 Controller 接收参数，调用 Service 完成业务逻辑，再由 Mapper 访问数据库，必要时还会结合 Redis、WebSocket、支付宝或 DeepSeek API 进行协同处理，最后将结果封装成统一的 `Result` 返回前端。

### 13.2 管理端请求链路
管理端请求链路和用户端类似，但它的登录方式更复杂，因为后台不是只有一个统一管理员，而是区分系统管理员、门店管理员和门店员工。后台登录成功后，会把 `admin-token`、`admin-user` 和 `admin-login-type` 保存到本地存储。前端页面路由先通过 `roles + staffRoles + permissionCodes` 做一次菜单级访问控制，后端 `JwtTokenInterceptor` 再根据接口 URI、当前登录角色和权限码做一次接口级拦截。也就是说，后台权限控制既发生在前端页面级，也发生在后端接口级，两层共同作用，保证权限模型真正生效。

### 13.3 典型业务联动链路
以预约业务为例，用户选择剧本和场次后发起下单，请求进入后端后先校验排期是否可用，再校验房间冲突，再重算金额，再处理 VIP 折扣和优惠券，然后写入订单、更新排期人数、必要时自动发起拼团并发送通知。支付成功后还会生成核销码，门店员工再基于核销码完成到店核销。这条链路串联了剧本、门店、排期、预约、支付、通知、员工协同和评价几个模块，是整个系统最核心的业务主线。

## 14. 三条最关键业务流程

### 14.1 预约下单流程
你答辩时最应该讲熟的是预约主流程。第一步，用户在剧本列表或门店详情页进入排期大厅。第二步，选择具体场次、人数和优惠券，进入确认预约页面。第三步，后端依据剧本价格和人数重新计算总价，并尝试叠加 VIP 折扣与优惠券抵扣，避免前端篡改金额。第四步，创建预约订单并更新场次人数。第五步，如果人数不足剧本要求人数，系统会自动创建或挂接拼团记录。第六步，支付成功后系统生成核销码并推送通知。第七步，到店后由门店员工完成核销，场次结束后由管理员工或 DM 标记完成，最后用户可提交评价并获得积分奖励。

### 14.2 VIP 购买与续费流程
VIP 购买流程同样值得讲，因为这里体现了项目的会员与营销设计。用户在 VIP 页面查看套餐详情后选择购买或续费，前端调用 `/api/vip/purchase` 或 `/api/vip/renew`，后端创建一条待支付的 `UserVip` 记录并生成订单号。支付成功后，系统会通过异步回调激活会员。当前项目已经修复了“续费后天数不叠加”的问题，激活时不再只查“当前生效会员”，而是查“最后一条已支付会员记录”，以它的结束时间为新的起始时间继续叠加。完成激活后，系统会同步更新用户的 `vipLevel` 和 `vipExpireTime`，并在前端 layout 和 VIP 页面中刷新到 `userStore`，确保页面显示和 AI 客服读取到的都是最新会员状态。

### 14.3 AI 客服流程
AI 客服流程是项目的创新亮点。用户在任意页面打开悬浮客服组件后输入问题，前端会先收集最近 10 轮消息历史，再把当前页面路径和当前用户的 VIP 等级一起传给后端。后端 `AIServiceImpl` 会先根据平台真实业务规则构造 `systemPrompt`，再调用 DeepSeek 官方接口完成回复。如果调用失败，前端会回退到本地知识库关键词匹配模式，保证对话不会中断。如果用户输入“转人工”等关键词，系统会创建人工客服会话并切换为人工模式，通过 WebSocket 实时通信完成后续消息收发。

## 17. 调试与排错入口

### 17.1 看登录和权限问题
如果出现“能看到页面但接口 403”或“登录后部分功能不能操作”，首先检查：
1. 前端本地存储中 `admin-login-type` 是否正确。
2. `admin-user` 中的 `staffRole` 和 `permissionCodes` 是否符合预期。
3. 后端 `JwtTokenInterceptor` 是否把 role、storeId、staffRole、permissionCodes 正确写入 `BaseContext`。
4. Controller 或 Service 中是否还有 scope 限制，例如 DM 只能看自己的预约。

### 17.2 看预约问题
如果预约失败，应优先看 `ReservationServiceImpl.create()`。这个方法里依次会做：排期校验、房间冲突校验、金额重算、VIP 折扣处理、优惠券处理、排期人数更新、拼团创建与通知发送。一般预约出问题，根因都在这条链路里。

### 17.3 看 VIP 问题
如果 VIP 等级显示不对或续费逻辑异常，应优先看 `VipServiceImpl` 和用户端 `layout/index.vue`、`vip/index.vue`。后端决定会员激活逻辑，前端决定会员状态是否同步到了 `userStore`。AI 客服读取的也是这里的 `vipLevel`。

### 17.4 看 AI 客服问题
如果 AI 回复不自然、没有上下文或像是固定模板，重点检查三处：
1. 前端 `AICustomerService.vue` 是否传了 `history`、`sessionId` 和 `context.userInfo.vipLevel`。
2. 后端 `AIServiceImpl.buildSystemPrompt()` 是否仍是项目熟悉版提示词。
3. `application.yml` 中的 `ai.provider`、`api-url` 和 `model` 是否指向 DeepSeek 官方配置。

## 18. 快速复习清单（答辩前必背）

### 18.1 你必须能脱口而出的 10 个点
1. 这个项目不是普通展示站，而是剧本杀门店数字化运营平台。
2. 技术架构是 Spring Boot 单体后端 + Vue 用户端 + Vue 管理端。
3. 后端核心依赖 MySQL、Redis、JWT、WebSocket、支付宝、DeepSeek。
4. 系统角色包括普通用户、系统管理员、门店管理员和门店员工。
5. 门店员工进一步分为 MANAGER、DM、CLERK，并配合 permissionCodes 控制权限。
6. 预约模块是系统主线，串联剧本、门店、排期、支付、核销、评价。
7. 价格由后端重算，并叠加 VIP 折扣和优惠券。
8. 拼团不是独立模块，而是和预约联动的人数补足机制。
9. AI 客服已经做成项目熟悉版，并支持转人工。
10. 当前亮点是业务闭环完整、权限设计细、营销和客服能力丰富。

### 18.2 你最好也能讲出来的 5 个优化点
1. 可以把当前单体架构进一步拆成更细的服务。
2. 可以把 AI 客服从提示词模式升级为真正的知识库/RAG 检索。
3. 可以把敏感配置改成环境变量或配置中心。
4. 可以补充更多自动化测试和性能测试。
5. 可以在推荐模块上继续增强用户画像能力。
