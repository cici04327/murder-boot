# 门店运营模块分析报告

## 一、门店生命周期与查询

### 1.1 门店实体（Store）
- **核心属性**：
  - 基础信息：名称、地址、电话、图片、描述
  - 营业时间：openTime、closeTime（LocalTime）
  - 地理位置：longitude、latitude (BigDecimal)
  - 评分：rating（BigDecimal）
  - 状态：status（1=营业中，0=已打烊）
  - 登录账号/密码：loginAccount、loginPassword（MD5加密）

### 1.2 生命周期管理
- **新增**：StoreController.add() → StoreServiceImpl.add()
  - 仅超级管理员可操作（requireSuperAdmin()）
- **查询**：
  - pageQuery()：支持名称模糊查询、分页
  - pageQueryAdvanced()：多条件查询（名称、地址、状态、评分范围）
    - 支持按评分/热度/距离排序
    - **距离排序**使用Haversine公式计算地心距离（单位：km）
  - getById()：单个查询，使用Redis缓存（key: `store:${id}`）
  - getDetailById()：详细信息（包含房间列表、评价统计）
  - listAll()：全量列表，支持storeId筛选
  - getHotStores()：热门门店（最新10条）
  - getRecommendedStores()：推荐门店（最新10条）
  - getNearbyStores()：基于经纬度的附近门店查询

- **更新**：
  - update()：更新门店信息，同时清除缓存
  - updateStatus()：修改营业状态
  - batchUpdateStatus()：批量修改状态
  - updateLoginPassword()：修改门店登录密码（验证旧密码）
  - updateStoreAccount()：更新登录账号（检重）
  - resetStorePassword()：重置为默认密码 123456

- **删除**：
  - delete()：单个逻辑删除
  - batchDelete()：批量逻辑删除

### 1.3 门店登录认证
- **端点**：POST /api/store/login
- **流程**：
  1. 根据 loginAccount 查询门店
  2. 验证门店状态（必须营业中）
  3. MD5密码校验
  4. 生成JWT令牌
     - USER_ID：-store.id（负值区分）
     - STORE_ID：store.id
     - ROLE：STORE_ADMIN
     - TTL：由JwtProperties.adminTtl决定
- **返回**：StoreLoginVO（包含token、门店基本信息）

### 1.4 统计信息（StoreStatisticsVO）
- 总门店数、营业/停业数
- 房间数（总数、可用数）
- 评分（平均分）、评价总数、好评数
- **权限隔离**：
  - 超级管理员：统计全部门店
  - 门店管理员（STORE_ADMIN）：仅统计自己门店

## 二、房间管理

### 2.1 房间实体（StoreRoom）
- **属性**：
  - storeId：所属门店
  - name：房间名称
  - type：房间类型（1=小房，2=中房，3=大房）
  - capacity：容纳人数
  - description：房间描述
  - status：状态（1=可用，0=不可用）
  - images：图片（@TableField(exist=false)，未持久化）

### 2.2 房间操作（StoreRoomController/StoreRoomServiceImpl）
- **查询**：
  - getById()：单个房间详情
  - listByStoreId()：按门店ID查询列表，按type升序
  - getAvailableRooms()：仅查询status=1的可用房间
- **增改删**：
  - add()、update()、delete()：CRUD操作
  - 无权限控制（默认管理员可操作）

### 2.3 与预约关联
- 房间在 script_schedule 中使用
- 房间利用率统计（StatisticsServiceImpl）：
  - 总排期数 / 被使用的排期数（current_players > 0）

## 三、员工与DM体系

### 3.1 员工实体（StoreEmployee）
- **核心属性**：
  - storeId、name、phone、position、joinDate、salary
  - position：1=店长，2=副店长，3=主持人（DM），4=服务员
  - loginAccount、loginPassword（MD5）
  - status：1=在职，0=离职
  - staffRole：后台角色（MANAGER/CLERK/DM）
  - permissionCodes：权限编码，逗号分隔
  - dmId：绑定的DM账号ID（主持人）

### 3.2 员工权限体系
**三层权限架构**：
| staffRole | 默认权限 | 应用场景 |
|-----------|---------|--------|
| MANAGER | reservation:view, :checkin, :complete, :assign_dm, refund:process, employee:manage, report:view | 店长（position=1,2） |
| DM | reservation:view, :complete, notification:view | 主持人（position=3） |
| CLERK | reservation:view, :checkin, notification:view | 其他员工（position=4） |

- **权限限制机制**（applyRolePermissionLimit）：
  - 员工选择的权限必须在该角色允许范围内
  - 防止权限提升攻击

### 3.3 员工管理（StoreEmployeeController/StoreEmployeeServiceImpl）
- **查询**：
  - pageQuery()：分页员工列表
  - listByStoreId()：按门店列表
  - getById()：单个员工
  - **权限要求**：SUPER_ADMIN/STORE_ADMIN 或 STORE_STAFF + employee:manage

- **增改删**：
  - add()：新增员工
    - 默认密码：123456（MD5加密）
    - 登录账号默认使用phone
    - 自动赋予角色和权限
  - update()：更新员工（保护已有密码、账号等）
  - delete()/batchDelete()：逻辑删除
  - updateStatus()：启用/禁用账号
  - resetPassword()：手动重置密码

- **登录流程**：
  - 端点：POST /api/store/employee/login
  - 验证流程：
    1. 查询账号（必须存在、状态=1）
    2. MD5密码校验
    3. 验证所属门店状态（必须营业）
    4. 生成JWT令牌
     - EMPLOYEE_ID、STORE_ID
     - ROLE：STORE_STAFF
     - STAFF_ROLE、PERMISSION_CODES
     - DM_ID（如果绑定了DM）
  - 返回：StoreEmployeeLoginVO

- **操作日志**：
  - 记录员工创建、更新、状态变更、密码重置等操作
  - 由StoreEmployeeOperationLogService处理

### 3.4 权限检查（StoreEmployeeController）
```
requireEmployeeManageAccess()：
  ├─ SUPER_ADMIN ✓
  ├─ STORE_ADMIN ✓
  └─ STORE_STAFF + employee:manage ✓

resolveStoreScope()：
  ├─ STORE_ADMIN/STORE_STAFF → 强制返回 BaseContext.getStoreId()
  └─ SUPER_ADMIN → 使用请求参数 storeId

assertEmployeeStoreScope()：
  ├─ 员工必须属于当前操作者的门店
  ├─ STORE_STAFF 不能查看/管理 MANAGER 级员工
  └─ 店长账号仅允许管理端维护
```

### 3.5 DM（主持人）体系（Dm实体与DmService）
- **核心属性**：
  - storeId：所属门店
  - name/avatar：艺名和头像
  - introduction：简介
  - styleTags：风格标签（逗号分隔，如：推理型,情感型,恐怖型）
  - rating：综合评分（BigDecimal，精度0.1）
  - gameCount：累计主持场次
  - status：1=在职，0=离职

- **DM查询接口**：
  - pageQuery()：分页查询（支持storeId、status筛选）
  - listByStoreId()：按门店列表（仅在职，按rating降序）
  - getById()：单个DM详情
  
- **DM操作**：
  - add()：新增DM
    - rating初始为0
    - gameCount初始为0
    - 不允许直接修改这两个字段
  - update()：更新基本信息（name、avatar、introduction、styleTags）
  - delete()：逻辑删除
  - updateStatus()：修改在职/离职状态
  - refreshRating()：根据评价自动计算评分
    - 公式：取所有已通过评价(status=2)的dmRating均值，精度0.1

- **DmVO特性**：
  - storeName：关联查询门店名称
  - styleTagList：将styleTags拆分为列表（前端展示）
  - statusName：1→"在职"，0→"离职"

- **与员工关系**：
  - 员工.dmId → DM.id（主持人类员工可绑定DM账号）
  - 用于区分员工的多重身份（员工+主持人）

## 四、统计与经营分析

### 4.1 统计接口（StatisticsController）

#### 4.1.1 概览数据（/api/statistics/overview）
**StatisticsOverviewVO**包含：
- 今日/昨日营业额、增长率
- 今日/昨日预约数、增长率
- 今日新增用户、累计用户数
- 在线门店数、总门店数
- 优惠券使用数、使用率
- 精选剧本数
- 用户满意度（%）

#### 4.1.2 图表数据（/api/statistics/charts?days=7）
**StatisticsChartsVO**包含：
- 营业额趋势（日期+金额）
- 预约来源分布（线上/电话/到店）
- 用户增长趋势（日期+数量）
- 会员等级分布（青铜/白银/黄金/钻石）

#### 4.1.3 排行榜（/api/statistics/rankings?limit=10）
**StatisticsRankingVO**包含：
- 门店营收排行（门店名、营收、预约数）
- 剧本热度排行（剧本名、预约数、评分）

#### 4.1.4 实时数据（/api/statistics/realtime?limit=10）
**StatisticsRealtimeVO**包含：
- 最新预约动态（用户、剧本、门店、时间）
- 今日热门剧本TOP5
- 最近优惠券使用记录

### 4.2 经营看板（/api/statistics/operation-board?days=30&storeId=X）

**核心指标**（返回Map格式）：

| 指标 | SQL逻辑 | 计算方式 |
|-----|--------|--------|
| **转化率** | 已支付订单数/总订单数 | % |
| **取消率** | status=4订单数/总订单数 | % |
| **退款率** | refund_status=2订单数/总订单数 | % |
| **复购率** | 多次购买用户数/所有支付用户数 | % |
| **房间利用率** | 使用的排期数/总排期数 | % |
| **平均客单价** | SUM(actual_amount)/COUNT(支付订单) | 元 |
| **总营收** | SUM(actual_amount) 支付订单 | 元 |
| **每日趋势** | 按日期分组统计预约数和营收 | 日期数组 |
| **剧本热度TOP8** | 按预约数排序脚本 | 脚本列表 |
| **新老客户分层** | 是否在时间范围前有预约记录 | 新客/复购 |

- **权限隔离**（resolveAdminScopedStoreId）：
  - SUPER_ADMIN：可查询任意storeId
  - STORE_ADMIN：强制返回BaseContext.getStoreId()
  - STORE_STAFF + report:view：返回所属门店

### 4.3 门店日报（/api/statistics/store-daily-report?storeId=X）

**维度统计**（返回Map格式）：
- 今日/昨日/本周/本月营收、预约数、支付单数
- 环比增长率（今日vs昨日）
- 平均客单价
- 每小时预约分布（热力图数据）
- 本周每日营收趋势
- 今日剧本热度TOP5
- 今日退款/取消数

### 4.4 统计数据源
- **主要表**：reservation_order（预约订单）
- **关联表**：
  - store：门店基础
  - script：剧本信息
  - user：用户信息
  - user_coupon：优惠券
  - script_schedule：排期表
  - script_review/store_review：评价表

## 五、权限/作用域特点

### 5.1 用户身份体系

```
┌─ 普通用户（User）
│  └─ 无后台权限
│
├─ 超级管理员（SUPER_ADMIN）
│  ├─ 无限制访问所有门店/员工/DM数据
│  ├─ 可进行全局配置更新
│  └─ 可重置密码、更新账号
│
├─ 门店管理员（STORE_ADMIN）
│  ├─ 来自：Store.storeLogin()
│  ├─ 权限范围：仅限自己门店
│  ├─ 强制约束：
│  │  └─ BaseContext.getStoreId() 固定返回该门店ID
│  └─ 可操作：员工、房间、查看营收数据
│
└─ 门店员工（STORE_STAFF）
   ├─ 来自：StoreEmployee.login()
   ├─ staffRole分层：MANAGER/CLERK/DM
   ├─ 权限编码：permissionCodes（逗号分隔）
   ├─ 权限示例：
   │  ├─ MANAGER：employee:manage, report:view, refund:process
   │  ├─ DM：reservation:view, reservation:complete
   │  └─ CLERK：reservation:view, reservation:checkin
   ├─ 作用域限制：
   │  ├─ 仅能查看自己门店（BaseContext.getStoreId()）
   │  ├─ MANAGER不能被STORE_STAFF查看/管理
   │  └─ STORE_STAFF访问统计需report:view权限
   └─ DM身份：员工.dmId绑定DM账号
```

### 5.2 作用域控制模式

#### 模式一：请求参数覆盖（resolveStoreScope）
```java
if (role == "STORE_ADMIN" || role == "STORE_STAFF") {
    return BaseContext.getStoreId();  // 强制覆盖
}
return requestedStoreId;  // 超级管理员使用请求参数
```

#### 模式二：数据验证（assertEmployeeStoreScope）
```java
if (employee.storeId != BaseContext.getStoreId()) {
    throw new BaseException("无权限操作该门店员工");
}
```

#### 模式三：权限查询（applyManagerVisibleScope）
```
STORE_STAFF + staffRole==MANAGER时，
查询结果自动过滤掉其他MANAGER级员工
```

### 5.3 BaseContext上下文信息
```java
- getRole()：SUPER_ADMIN / STORE_ADMIN / STORE_STAFF
- getStoreId()：当前用户关联的门店ID（null表示未绑定）
- getStaffRole()：后台角色（MANAGER/CLERK/DM）
- getPermissionCodes()：权限编码字符串
- getDmId()：DM账号ID（如果有）
```

### 5.4 权限检查特性

| 场景 | 检查点 | 机制 |
|------|-------|------|
| 查看员工列表 | requireEmployeeManageAccess() | SUPER_ADMIN OR STORE_ADMIN OR (STORE_STAFF+employee:manage) |
| 查看统计数据 | resolveAdminScopedStoreId() | 权限+角色双重验证 |
| 访问特定员工 | assertEmployeeStoreScope() | 门店隔离+职级隔离 |
| 更新员工角色 | rejectManagerTargetForStaff() | STORE_STAFF不能触碰MANAGER级 |
| 更新门店状态 | requireSuperAdmin() | 仅SUPER_ADMIN |

### 5.5 安全特性

1. **密码保护**：
   - 所有密码均MD5加密存储
   - 登录时重新验证加密值
   - 重置密码时需原密码验证（员工除外）

2. **账号唯一性**：
   - validateLoginAccountUnique()：员工登录账号全局唯一
   - 检查时排除自己和已删除的记录

3. **级联权限**：
   - 员工创建时自动赋予角色对应权限
   - 权限更新时只允许选择角色允许范围内的权限

4. **操作日志**：
   - 记录员工增删改查操作
   - 记录权限变更细节
   - 便于审计和问题追踪

## 七、关键设计模式

### 7.1 作用域设计
- **两段式处理**：先通过权限检查，再强制覆盖storeId
- **防止权限提升**：每次操作都二次验证员工所属门店

### 7.2 缓存策略
- StoreServiceImpl使用@Cacheable和Redis双重缓存
- 更新时使用@CacheEvict清除所有相关缓存
- 热点数据（热门门店）利用缓存提升性能

### 7.3 权限编码设计
- 权限粒度细：reservation:view / :checkin / :complete / :assign_dm
- 按角色预设，按岗位自动分配，支持手动调整
- 权限码用逗号分隔存储在单字段，查询时split()解析

### 7.4 统计查询优化
- 使用JdbcTemplate直接SQL查询（性能考虑）
- safeQueryInt/safeQueryBig：异常处理和兜底值
- 日期范围查询使用LocalDateTime.MIN/MAX精确时间边界

### 7.5 DM独立模块
- DM既可作为员工岗位，又可独立管理
- 评分通过Review表的dmRating字段自动计算
- gameCount与预约订单关联
